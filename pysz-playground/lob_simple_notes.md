# 深交所逐笔还原规则与 `lob_simple` 设计思路

这份说明只服务于一个目标：解释如何用逐笔委托和逐笔成交，还原一个可理解的限价订单簿。

参考资料，以深交所数据接口页面的最新版本为准：

- 深圳证券交易所 STEP 行情数据接口规范 Ver1.16：https://investor.szse.cn/marketServices/technicalservice/interface/P020240531618144293021.pdf
- 深圳证券交易所数据接口页面：https://www.szse.cn/marketServices/technicalservice/interface/

## 一、深交所逐笔规则的工程总结

深交所逐笔行情可以看成一条按时间发生顺序排列的报文流。报文主要有两类：

- 逐笔委托：一笔新订单进入市场。
- 逐笔成交：一笔订单被成交，或一笔订单被撤销。

官方规范里，逐笔委托和逐笔成交会按照发生顺序进入同一套逐笔行情序列，同一个频道内的消息记录号连续编号。工程上这意味着：重建 LOB 时，最重要的前提是按正确顺序逐条处理报文。

### 逐笔委托

逐笔委托报文提供一笔订单的基本信息：

- 订单编号：后续成交或撤单会用它引用这笔订单。
- 买卖方向：买单或卖单。
- 委托价格：建议转换成整数 tick，不要直接用 float 做价格档 key。
- 委托数量：订单初始数量。
- 订单类型：限价、市价等。

对 LOB 来说，一笔限价委托会做两件事：

1. 记录到 `orders`，方便未来通过订单号找到它。
2. 加到买盘或卖盘对应价格档。

市价单通常不作为静态深度挂在盘口上，所以在这个教学版里只进入订单池，不增加价格档深度。

### 逐笔成交与撤单

逐笔成交报文会携带买方委托索引、卖方委托索引、成交数量，以及执行类型。

执行类型是重建逻辑里的关键。真实 STEP 接口里逐笔成交执行类型使用 `F=Trade`、`4=Cancelled`；本项目的教学 CSV 为了更直观，映射成了 `T=Trade`、`C=Cancel`。

- 成交：同时减少被引用的买方订单和卖方订单。
- 撤单：减少被撤销的那一侧订单。
- 委托索引为 0 时，表示没有对应委托，代码里用 `NO_ORDER = 0` 跳过。

无论是成交还是撤单，本质上都是对存活订单的扣量：

1. 找到订单。
2. 减少订单剩余数量。
3. 如果它是限价单，同步减少对应价格档数量。
4. 如果订单剩余数量为 0，从订单池删除。

所以代码里把公共逻辑收进 `_reduce_order()`，成交和撤单只是告诉它要扣哪几个订单号。

### 数据字段和本项目 CSV 的对应关系

`test_data/lob_events.csv` 使用的是便于阅读的教学字段：

| CSV 字段 | 含义 |
| --- | --- |
| `EventType` | `O` 表示委托，`E` 表示执行报告 |
| `ExecType` | `T` 表示成交，`C` 表示撤单 |
| `OrderNo` | 新委托的订单编号 |
| `BidOrderNo` | 成交或撤单涉及的买方订单编号 |
| `AskOrderNo` | 成交或撤单涉及的卖方订单编号 |
| `Price` | 价格，进入 LOB 前转成整数 tick |
| `Volume` | 数量 |
| `Side` | `B` 买，`S` 卖 |
| `OrderType` | `L` 限价，`M` 市价 |

真实深交所接口中的取值更偏底层，例如买卖方向是 `1/2`，订单类型包括 `1/2/U`，成交执行类型是 `F/4`。`lob_simple.py` 故意把这些映射留在很薄的一层函数里，方便替换真实数据源。

## 二、`lob_simple.py` 的设计思路

`lob_simple.py` 不是为了极致性能，而是为了把 LOB 还原的核心原理讲清楚。

### 1. 状态只保留两类东西

订单簿重建需要的状态其实很少：

```python
self.orders = {}
self.bids = defaultdict(int)
self.asks = defaultdict(int)
```

`orders` 是订单池：`order_no -> Order`。

`bids` 和 `asks` 是价格档聚合：`price -> total_volume`。

只要这两个状态保持一致，就能随时回答：

- 某笔订单还剩多少量。
- 某个价格档有多少量。
- 当前买一、卖一在哪里。

### 2. `Order` 只描述一笔存活订单

```python
@dataclass
class Order:
    price: int
    volume: int
    side: int
    order_type: int
```

它不关心 pandas，也不关心报文格式，只表示一笔订单当前还剩多少量，以及它挂在哪一边、哪一个价位。

这里保留了整数 enum：

```python
BUY = 1
SELL = -1
LIMIT = 1
MARKET = 2
```

这样比字符串更接近底层编码，也比 Numba 版本的类型声明更适合讲原理。

### 3. 价格用整数 tick

价格进入订单簿前会做：

```python
int(round(float(price) * PRICE_SCALE))
```

例如 `10.10` 会变成 `101000`。这样避免把浮点数作为字典 key，防止同一个价格因为浮点误差落到不同档位。

### 4. 一行 pandas 记录就是一个报文

教学版的主入口是：

```python
for row in df.itertuples(index=False):
    process_message(book, row)
```

`process_message()` 只做三件事：

1. 看 `EventType` 判断是委托还是执行报告。
2. 把 CSV 里的 `B/S/L/M/T/C` 转成内部整数符号。
3. 调用 `book.add_order()`、`book.trade()` 或 `book.cancel()`。

这样可以把“报文解释”和“订单簿状态变化”分开。

### 5. 成交和撤单复用同一套扣量逻辑

成交会扣订单，撤单也会扣订单。区别只是报文里引用的是哪一侧订单。

因此：

```python
def trade(...):
    self._reduce_order(...)

def cancel(...):
    self._reduce_order(...)
```

真正维护一致性的地方只有 `_reduce_order()`：

- 订单不存在，报错。
- 扣量超过剩余量，报错。
- 限价单同步扣价格档。
- 剩余量为 0，删除订单。

这让调试路径很短：如果盘口数量错了，大概率看 `_reduce_order()` 就够了。

## 三、这版故意没有做的事

为了可读性，`lob_simple.py` 暂时不处理这些生产级问题：

- 多证券、多频道拆分。
- 逐笔序号连续性检查和重传。
- 集合竞价、停牌、收盘等交易阶段细节。
- 更复杂的订单类型，比如本方最优。
- 用有序树、堆或跳表优化买一卖一查询。
- 用快照行情校验还原结果。

这些都可以在理解核心状态机之后再加。当前版本的价值是把 LOB 还原压缩成一句话：

> 委托增加存活订单和价格档；成交或撤单减少存活订单和价格档；订单量归零时删除。
